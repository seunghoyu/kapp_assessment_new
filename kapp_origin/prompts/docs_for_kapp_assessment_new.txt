# kapp_assessment_new 프로젝트 문서화용 통합 프롬프트

아래 지시를 그대로 복사해, 문서 작성자(또는 AI)에게 입력하여 사용하세요.
---

========== 여기부터 복사 ==========

## 목적
- **기준 프로젝트**: kapp_origin (기존 HTML/CSS/JS 기반 전체 서비스)
- **대상 프로젝트**: kapp_assessment_new (Next.js 기반, 현재는 관리자 대시보드 위주로 진행 중)
- kapp_origin의 docs에 있는 **01~05번 프로젝트 브리프 문서**와 같은 형식으로, **kapp_assessment_new 전용** 문서 세트를 만들어 달라.

## 프로젝트 관계 및 주요 변경점 (반드시 반영할 것)

1. **기준(origin) vs 대상(new)**
   - **origin**: HTML/CSS/JS, 다수 페이지(홈·KAPP 진단·마이 대시보드·교육 큐레이션·나의 성장·알림·관리자), 스크롤 위주, 데이터는 JS 내 인라인/목 데이터.
   - **new**: Next.js(App Router), 사이드바·헤더 등 레이아웃 갖춤, 데이터 표시 간소화(관리자·학습자 모두), 키 컬러 기반 디자인, **실제 더미 데이터는 JSON 폴더(data/)에서 로드**.

2. **레이아웃**
   - new에서는 스크롤만 있는 형태가 아니라 **사이드바 + 헤더** 구조로 변경됨. 문서에 이 레이아웃 전제를 명시할 것.

3. **데이터 표시**
   - 관리자·학습자 모두 **데이터 노출을 간소화**하는 방향. “어디를 줄였는지” “어떤 블록만 남겼는지”를 문서에서 구체적으로 적을 것.

4. **데이터 소스**
   - new에서는 **data/ 폴더 내 JSON 파일**을 더미 데이터로 사용. (예: summaryStats, departmentPerformance, employees, benchmark, industryBenchmarks, progressPrograms, insights, heatmap, riskManagement, roi, strategyMapper, highPerformer 등) HTML/CSS/JS에서 쓰이던 인라인 데이터는 이 JSON 구조로 이전된 것으로 기술할 것.

5. **기술 스택**
   - HTML/CSS/JS → **Next.js**(+ Tailwind 등 현재 new에서 사용 중인 스택)로 변경. 문서에는 “현재 new 프로젝트는 Next.js 기반”이라고 명시할 것.

## 산출물 요청 (총 6개 문서 + 1개 통합)

다음 문서들을 **kapp_assessment_new** 프로젝트 루트의 **docs/** 폴더에 생성해 달라.  
(기존 origin의 01~05 형식·목차 스타일을 참고하되, **내용은 전부 new 기준**으로 작성.)

---

### 문서 1: 기획자 관점 (예: 01_project_brief_product_new.md)
- new 프로젝트의 **비전·목표·사용자 역할** (관리자 vs 학습자).
- **사용자 여정·데이터 흐름**: new에서의 페이지/라우팅, 데이터가 어디서(JSON) 오고 어디로 가는지.
- **페이지별 기능·목적**: new에 실제로 존재하는 페이지와, origin에는 있으나 new에는 아직 없는 페이지를 구분해서 기술.
- **구현 상태 반영**: “현재 new에는 관리자 대시보드(대시보드·역량 등)만 구현되어 있고, KAPP 진단·학습자 대시보드·교육 큐레이션·나의 성장·알림 페이지는 미구현”임을 명시하고, 각 미구현 영역의 기획적 목적만 간단히 정리.

---

### 문서 2: 프론트엔드 관점 (예: 02_project_brief_frontend_new.md)
- new의 **페이지·라우팅 구조** (app/ 디렉터리 기준).
- **공통 레이아웃**: 사이드바, 헤더, 메인 영역 역할과 조합 방식.
- **데이터 소스**: 어떤 화면이 **data/ 내 어떤 JSON**을 참조하는지 매핑. (예: 대시보드 KPI → summaryStats, 부서 성과 → departmentPerformance.json 등)
- **구현된 UI·로직**: 대시보드·역량(벤치마크)·차트·필터·테이블/뷰 전환 등 **이미 구현된 부분**만 상세히. (컴포넌트 이름은 필요 시 최소한만)
- **미구현 부분**: 진단·학습자 대시보드·교육·나의 성장·알림 등은 “미구현”으로 표시하고, origin의 해당 화면이 어떤 데이터·어떤 동작을 했는지 요약만 적어 “나중에 구현 시 참고” 수준으로 정리.
- **상태·이벤트**: 구현된 페이지에서 필터/탭/뷰 전환 시 어떤 상태가 바뀌고 어떤 데이터가 다시 쓰이는지.

---

### 문서 3: 백엔드 관점 (예: 03_project_brief_backend_new.md)
- new는 **아직 API 서버가 없다**는 전제. 현재는 **정적 JSON(data/)** 을 프론트에서 직접 읽는 구조임을 명시.
- **데이터 엔티티**: new의 data/ JSON 파일별로 “어떤 엔티티(요약 통계·부서 성과·임직원·벤치마크·진행 프로그램·인사이트·히트맵·리스크·ROI·전략 매핑·고성과자 등)를 나타내는지” 정리.
- **연동 시 API 설계 방향**: 추후 백엔드를 붙일 때, 위 엔티티들을 어떤 API(조회·필터·페이지네이션)로 나눌지 제안. (origin의 03_project_brief_backend.md와 유사한 톤으로, but new의 data/ 구조에 맞게.)
- **구현된 것 vs 미구현**: “현재 new에서 사용 중인 JSON 목록”과 “origin에는 있으나 new에 JSON/로직이 아직 없는 데이터”를 구분해서 기술.

---

### 문서 4: 마케터 관점 (예: 04_project_brief_marketer_new.md)
- new 서비스의 **타깃**(관리자·학습자), **핵심 메시지·포지셔닝** (레이아웃·간소화·키 컬러 등 변경점을 메시징에 반영).
- **페이지별 메시징·CTA**: 현재 구현된 관리자 대시보드 위주로. 미구현 페이지는 “예정 메시지/CTA” 수준으로만.
- **지표·전환**: 관리자 대시보드에서 어떤 액션(필터·탭·다운로드 등)을 전환 지표로 볼 수 있는지.
- **브랜딩·키 컬러**: new에서 적용 중인 디자인(키 컬러 등)을 마케팅 관점에서 어떻게 활용할지 한 줄씩 정리.

---

### 문서 5: 데이터 분석가 관점 (예: 05_project_brief_data_analyst_new.md)
- new의 **data/ JSON 스키마 요약**: 파일별로 “어떤 필드가 있고, 어떤 용도인지” (변수명 나열보다는 “역할·의미” 위주).
- **데이터 흐름**: 어떤 JSON이 어떤 화면/컴포넌트에서 쓰이는지 표로 정리.
- **정합성·확장**: 현재 JSON만으로 관리자 대시보드가 동작하는지, 부족한 필드나 추후 벤치마크·조정 계수 등을 넣을 때 어떤 JSON을 확장하면 되는지 제안.
- **origin과의 대응**: origin에서 쓰이던 더미 데이터(부서·스킬·벤치마크·전략 등)가 new의 어떤 JSON과 대응되는지 매핑 표 하나.

---

### 문서 6: 구현 현황 및 미구현 구현 가이드 (예: 06_implementation_status_and_guide_new.md)
이 문서는 **반드시 다음 4개 섹션**을 포함할 것.

#### 6.1 구현 현황 요약 (origin → new)
- **표 형태**로 정리 권장.
  - 열: 기능/화면 이름, origin 존재 여부, new 구현 여부, 비고(라우트·데이터 소스 등).
  - 행: 홈, KAPP 진단, 마이 대시보드(학습자), 교육 큐레이션, 나의 성장, 알림·설정, **관리자 대시보드**(요약 KPI, 부서별 차트, 역량/벤치마크, 진행 현황, 임직원 테이블, 인사이트, 스킬 갭 히트맵, 업계 벤치마킹 레이더, 전략–스킬 매핑, ROI 계산기, 고성과자, 스킬 리스크 등).
- “현재 new에는 관리자 대시보드 관련 라우트·컴포넌트·data/ JSON만 있고, 나머지(진단·학습자 대시보드·교육·나의 성장·알림)는 미구현”임을 명확히 구분할 것.

#### 6.2 new에서 이미 구현된 부분 상세
- **라우팅**: app/page.tsx, app/admin, app/dashboard, app/dashboard/competency 등 실제 경로와 역할.
- **레이아웃**: Sidebar, Header, 메인 영역 구성. (필요 시 layout.tsx 구조 언급.)
- **컴포넌트·데이터**: 대시보드 KPI 카드, 부서 성과 차트, 부서 스킬 차트/필터, 벤치마크 섹션, 사용 중인 JSON 파일 목록 등 “이미 있는 것”만 나열.
- **스타일**: 키 컬러·Tailwind 사용 여부 등 DESIGN_SYSTEM 또는 globals.css 기준 요약.

#### 6.3 미구현 부분 구현 방향 (어떻게 구현할지)
- **KAPP 진단**: origin의 assessment-kapp 흐름(정보 입력 → 문항 → 제출 → 결과 저장)을 Next.js 페이지/API Route로 옮기는 방향. 데이터는 추후 API 또는 data/에 assessment 관련 JSON 추가.
- **학습자 마이 대시보드**: origin의 dashboard-kapp 대응. 진단 결과(또는 JSON) 기반 DNA·KAPP 점수·레이더·시장 포지션 등. new에서는 data/에 결과 예시 JSON을 두고 해당 페이지에서 읽는 방식 제안.
- **교육 큐레이션**: 과정 목록·필터·추천·상세. data/에 courses.json 등 추가 후, 목록/상세 페이지와 컴포넌트 구현 방향.
- **나의 성장**: 1% 팁·AI 멘토·인증서. origin 로직을 참고해 페이지·상태·로컬 저장 또는 API 연동 방향만 간단히.
- **알림·설정**: 설정 토글·알림 권한 등. 페이지 하나 + 설정 JSON 또는 API 제안.
- 각 항목은 “무엇을 만들고, 어떤 데이터(JSON/API)를 쓰면 되는지” 2~3문단 수준으로 작성.

#### 6.4 origin에 있던 기능·로직 중 new 문서화 시 참고할 목록
- origin의 **페이지·기능 목록**을 짧게 나열하고, “위 6.3의 미구현 구현 시 origin의 해당 HTML/JS 로직을 참고할 것”이라고 명시. (예: 스킬 갭 히트맵 로직, 벤치마킹 레이더 계산, 전략 매핑·ROI 계산 등.)

---

### 문서 7 (선택): 데이터 레퍼런스 통합 – new 반영 (예: 07_data_reference_integration_new.md)
- origin의 **05_data_reference_integration.md** 내용을 전제로, “new 프로젝트에서 데이터 레퍼런스(KLI·KRIVET·조정 계수·벤치마크·JSON)를 어떻게 반영할지”만 추가 정리.
- new의 **data/** 에 조정 계수·벤치마크용 JSON을 둘 경우 파일명·용도 제안.
- 역할별(기획·프론트·백엔드·마케터·데이터 분석가) “new에서 데이터를 어떻게 접근할지” 한 줄씩이라도 적어 줄 것.

## 역할별 공통 요청
- **기획자, 프론트엔드, 백엔드, 마케터, 데이터 분석가** 5개 관점이 **각각 문서에 반영**되어야 함. (문서 1~5가 역할별 대응, 문서 6·7은 통합.)
- 문서 안에서 “기획자 관점에서는 …”, “프론트엔드 관점에서는 …”처럼 **역할을 명시한 소제목**을 넣어도 됨.
- **변수명·클래스명**은 꼭 필요할 때만 최소한으로. **로직·데이터 흐름·구현 여부·구현 방향** 위주로 작성할 것.
- new 프로젝트의 **실제 폴더 구조(app/, components/, data/)** 와 **실제 구현된 페이지·컴포넌트**를 기준으로 작성하고, 없는 기능은 “미구현” 또는 “구현 방향”으로만 기술할 것.
